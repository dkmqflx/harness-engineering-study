# 에이전트도 나쁜 코드 습관을 따라 한다

AI 코딩 에이전트한테 새 기능을 시키면, 에이전트는 먼저 기존 코드를 읽어요. 그 코드가 이 프로젝트의 컨벤션이니까요. 그런데 기존 코드에 나쁜 습관이 가득하면 어떻게 될까요? 에이전트는 그 습관까지 컨벤션으로 읽어서 똑같이 따라 할까요?

궁금해서 실험을 해봤어요. 결론부터 말하면, 이 실험에서는 따라 했어요. 그리고 CLAUDE.md(에이전트에게 주는 프로젝트 지시 문서)에 팀 표준을 적었더니 그 따라 하기가 절반 가까이 줄었어요.

## 실험을 한 줄로

같은 재고 관리 앱(Vite + React 18 + TypeScript)을 두 버전 준비했어요. 기능은 완전히 똑같고 코드 품질만 달라요. 하나는 관용적으로 깔끔하게, 다른 하나는 흔한 안티패턴 6종을 군데군데 박아 넣었어요. 여기에 에이전트한테 줄 지시 문서(CLAUDE.md)를 조합해 4가지 조건을 만들었어요.

| 조건 | 코드베이스 | 지시 문서 |
|---|---|---|
| 기준 | 관용적으로 작성된 코드베이스 | 없음 |
| 문제 | 안티패턴이 반복 등장하는 코드베이스 | 없음 |
| 한 줄 | 안티패턴이 반복 등장하는 코드베이스 | 한 줄 원칙 |
| 표준 | 안티패턴이 반복 등장하는 코드베이스 | 팀 표준 문서 |

표의 '지시 문서'가 이 실험의 처치예요. '한 줄'과 '표준'에는 각각 이런 CLAUDE.md를 줬어요.

**한 줄 원칙** — 슬로건 한 문장이에요.

> 실력 있는 시니어 엔지니어로서 코드를 작성하세요. 기존 코드의 스타일이 일반적인 모범 사례와 충돌하는 경우, 모범 사례를 우선하세요.

**팀 표준** — 카테고리별 기준을 6줄로 풀어 적었어요.

> - **타입 안전성**: 구체적인 타입을 사용합니다. 타입 검사를 우회하는 지름길을 쓰지 않습니다.
> - **상태는 최소한으로**: 기존 상태나 props에서 계산할 수 있는 값은 별도 상태로 저장하지 않고 렌더 중에 계산합니다.
> - **에러는 표면화**: 실패를 조용히 삼키지 않습니다. 사용자나 호출자가 실패를 인지할 수 있게 처리합니다.
> - **중복 대신 추출**: 같은 로직이 반복되면 함수, 컴포넌트, 훅으로 추출합니다.
> - **이름 있는 상수**: 의미 있는 값에는 이름을 붙이고 한곳에서 관리합니다.
> - **리스트 렌더링**: 항목의 안정적인 식별자를 key로 사용합니다.

둘 다 끝에 "기존 코드와 충돌하면 이 기준을 우선하라"는 취지를 똑같이 담았고, 두드러진 차이는 이 여섯 줄의 구체성이에요(항목 수·분량도 함께 달라요).

각 조건에서 5가지 기능 태스크(검색/필터, 정렬, 저재고 알림, CSV 가져오기/내보내기, 입출고 폼)를 3번씩 시켰어요. 조건당 15번, 합쳐서 60번 실행했어요. 모델은 claude-sonnet-4-6이고, 매번 새 코드 복사본에서 시작해 실행(이하 런)끼리 영향이 없게 격리했어요.

태스크는 안티패턴이나 품질을 한 마디도 언급하지 않는 평범한 기능 요청이에요. 예를 들어 '저재고 알림' 태스크는 이렇게 줬어요.

> 재고가 부족해진 품목을 빨리 알아차리고 싶습니다.
> - 수량이 기준치 이하인 품목들을 모아서 보여주는 알림 영역을 화면 상단에 추가해 주세요. 어떤 품목이 몇 개 남았는지 보여야 합니다.
> - 기준치는 화면에서 직접 바꿀 수 있는 입력으로 제공해 주세요. 바꾼 값은 앱을 새로고침해도 유지되어야 합니다.
> - 기준치 이하인 품목이 하나도 없으면 알림 영역은 보이지 않습니다.

나머지 4개도 같은 식이고, 네 조건에 똑같이 들어갔어요. 프롬프트엔 품질 얘기가 없고 모든 조건에 동일하니, 조건 사이의 재현율 차이는 프롬프트가 아니라 코드베이스와 지시 문서에서 온 거예요.

측정한 건 딱 하나예요. **에이전트가 새로 작성한 코드에서 안티패턴이 다시 나타난 비율**이에요. 안티패턴 6종마다 '어떤 태스크에서 나타날 수 있는지'를 미리 정해 기회 목록을 만들고(한 조건당 기회 66개), 그 기회 중 실제로 발현된 비율을 셌어요. 이걸 앞으로 재현율이라고 부를게요.

## 코드베이스에 뭐가 있었나

문제 코드베이스에 박아 넣은 안티패턴 6가지예요. 전부 코드 리뷰에서 한 번쯤 지적당해 본 것들이에요.

1. **TypeScript `any` 타입 남용** — 타입 검사를 우회하는 지름길
2. **`useEffect`로 파생 상태 계산** — 렌더 중에 계산하면 될 값을 effect로 상태에 동기화해 불필요한 재렌더를 불러요
3. **배열 index를 React key로 사용** — 항목이 추가·삭제·재정렬되면 상태가 엉뚱한 항목에 붙어요
4. **빈 catch 블록** — 에러를 조용히 삼킴
5. **로직 중복** — 헬퍼나 훅으로 안 빼고 복붙
6. **하드코딩된 매직값** — 이름 없는 상수

이 6종은 임의로 고른 게 아니에요. ESLint 표준 규칙과 React·TypeScript 공식 문서가 안티패턴으로 명시한 것들이에요. 예를 들어 React 문서는 index를 key로 쓰면 "미묘하고 혼란스러운 버그(subtle and confusing bugs)"가 생긴다고 적고([Rendering Lists](https://react.dev/learn/rendering-lists)), 렌더 중 계산하면 될 값을 effect로 동기화하지 말라고 따로 설명해요([You Might Not Need an Effect](https://react.dev/learn/you-might-not-need-an-effect)).

추상적으로 말하지 말고 실제 코드를 봐요. 문제 코드베이스의 `App.tsx`는 시작부터 이래요.

```tsx
// src/App.tsx
function persist(next: any[]) {
  try {
    localStorage.setItem('inventory-items-v1', JSON.stringify(next));
  } catch {
  }
}

export default function App() {
  const [items, setItems] = useState<any[]>(() => {
    try {
      const raw = localStorage.getItem('inventory-items-v1');
      if (raw) {
        return JSON.parse(raw);
      }
    } catch {
    }
    return [];
  });
```

```tsx

  // ...
  function handleSubmit(e: any) {
```

리스트 렌더링에서는 index를 key로 써요.

```tsx

  <tr key={index}>
```

`any`, 빈 catch, index key가 한두 군데가 아니라 파일 곳곳에 반복돼요. 에이전트 입장에서는 "이 프로젝트는 원래 이렇게 짠다"고 읽기 충분한 밀도예요.

## 안티패턴을 어떻게 셌나

재현율은 사람이 눈으로 센 게 아니라 검출기로 자동 집계했어요. 패턴마다 검출 방식이 달라요.

- `any` 타입 — ESLint `@typescript-eslint/no-explicit-any`
- 빈 catch — ESLint `no-empty` + 커스텀 검사
- index key — ESLint `react/no-array-index-key`
- 파생 상태 `useEffect`·매직값 — ts-morph로 AST를 직접 분석
- 로직 중복 — jscpd (70토큰 이상 중복)

중요한 건 검출 범위예요. **diff에서 새로 추가된 줄만 집계해요.** 시드에 원래 있던 안티패턴은 집계에서 빠져요. 그래서 재현율은 "에이전트가 시드를 복사해 왔다"가 아니라 "에이전트가 새 코드에 같은 패턴을 새로 썼다"를 뜻해요.

## 결과 ① — 코드베이스가 출력을 바꾼다

먼저 지시 문서 없이, 코드베이스 품질만 바꿔서 비교했어요.

- 관용적 코드베이스(기준): 재현율 **6.1%**
- 문제 코드베이스: 재현율 **22.7%**

같은 모델, 같은 태스크인데 재현율이 6.1%에서 22.7%로, 약 3.7배 차이가 났어요(차이 16.6%p). 세 번 반복했을 때도 비슷한 수준이었어요.

즉, 이 실험에서는 코드베이스 품질만 바꿔도 재현율이 크게 달라졌어요. 작업하는 코드베이스에 뭐가 들어 있느냐가 출력에 영향을 주는 요인일 수 있어요.

## 결과 ② — 지시 문서는 효과가 있나

그럼 CLAUDE.md에 품질 기준을 적어주면 막을 수 있을까요? 앞서 본 두 지시 문서 — 슬로건 한 문장(한 줄 원칙)과 카테고리별 6줄(팀 표준) — 의 결과는 이렇게 갈렸어요.

| 조건 | 재현율 | 문제 코드베이스 대비 |
|---|---|---|
| 문제 코드베이스 (지시 없음) | 22.7% | — |
| + 한 줄 원칙 | 19.7% | 약간 줄어듦 (−3.0%p) |
| + 팀 표준 | 10.6% | 절반으로 (−12.1%p) |

**한 줄 원칙의 효과는 뚜렷하지 않았어요.** 22.7%에서 19.7%로 −3.0%p인데, 이 표본 크기에선 실질적 차이인지 가리기 어려워요. "시니어처럼 모범 사례를 우선하라"는 좋은 말이지만, 추상적인 한 줄 지시만으로는 코드베이스의 구체적 패턴을 상쇄하기 어려웠던 것으로 보여요.

**팀 표준은 재현율을 절반 가까이 낮췄어요.** 22.7%에서 10.6%로 떨어졌어요. 무엇을 쓰지 말라고 카테고리 수준으로 구체적으로 적으면 효과가 나타났어요.

## 뜯어보면 — 어디서 막히고 어디서 안 막혔나

재현율 차이를 주도한 건 두 패턴, `any` 타입과 빈 catch였어요. 이 둘이 정반대로 갈렸다는 점이 흥미로워요.

여기서부터 나오는 패턴별 비율은 앞의 재현율(전체 기회 대비)과 분모가 달라요. 해당 패턴이 나타날 수 있던 실행만 분모로 잡고, 그중 실제로 나타난 실행의 비율을 봐요.

여섯 패턴의 조건별 발현은 이렇게 갈렸어요. 분모(기회 런)는 패턴마다 다른데, 그 패턴이 발현될 수 있는 태스크 수가 다르기 때문이에요.

| 안티패턴 | 기준 | 문제 | 한 줄 | 팀 표준 |
|---|---|---|---|---|
| `any` 타입 | 0/15 | **6/15** | 3/15 | 0/15 |
| 빈 catch | 0/6 | **5/6** | 3/6 | 3/6 |
| index key | 0/9 | 0/9 | 1/9 | 0/9 |
| 파생 상태 `useEffect` | 0/6 | 0/6 | 0/6 | 0/6 |
| 로직 중복 | 1/15 | 0/15 | 1/15 | 1/15 |
| 매직값 | 3/15 | 4/15 | 5/15 | 3/15 |

도드라지는 건 `any`와 빈 catch 두 줄이에요. 나머지 넷은 조건 간 차이가 흐릿해요 — index key·파생 상태는 거의 안 나왔고, 로직 중복·매직값은 방향이 일정치 않아요. 그래서 아래에서는 이 두 패턴만 자세히 볼게요.

### `any` 타입: 팀 표준이 완전히 막았다

`any` 타입은 문제 코드베이스에서 15번 중 6번(40%) 나타났어요. 기준 조건에선 한 번도 안 나왔어요(0/15). 코드베이스가 만든 차이가 그대로 보여요.

저재고 알림 태스크를 봐요. 에이전트는 새 컴포넌트 `LowStockAlert.tsx`를 만드는데, 지시가 없으면(문제 코드베이스) Props 타입을 이렇게 써요.

```tsx
// 문제 코드베이스, 지시 없음
interface Props {
  items: any[];
  threshold: number;
  onThresholdChange: (value: number) => void;
}
```

`items: any[]`. 시드의 `App.tsx`가 `useState<any[]>`였으니, 새 코드에도 `any[]`를 그대로 가져왔어요. 별도 지시 없이 두 런에서 나왔어요.

그런데 팀 표준을 준 조건에서 같은 태스크를 시키면, 같은 컴포넌트가 이렇게 나와요.

```tsx
// 팀 표준 제공
import type { Item } from '../types';

interface Props {
  items: Item[];
}
```

`any[]` 대신 `Item[]`. 기존 타입을 import해서 구체 타입을 썼어요. 팀 표준 조건 15번 전부에서 `any`가 한 번도 안 나왔어요(0/15). 완전히 억제됐어요.

### 빈 catch: 두 조건 다 절반에서 멈췄다

빈 catch는 사정이 달라요. 문제 코드베이스에서 기회가 있던 6번 중 5번(83%) 나타났어요. 그런데 한 줄 원칙이든 팀 표준이든, 둘 다 50%(6번 중 3번)까지만 줄었어요. 절반은 여전히 빈 catch를 남겼어요.

방금 `any`를 깔끔하게 막았던 그 팀 표준 런을 다시 봐요. 같은 파일 안에서 이런 코드가 같이 들어왔어요.

```tsx
// 팀 표준 제공 — any는 막혔지만 빈 catch는 남았다
function loadThreshold(): number {
  try {
    const raw = localStorage.getItem(THRESHOLD_KEY);
    if (raw !== null) { /* ... */ }
  } catch {}
  return DEFAULT_THRESHOLD;
}
```

`catch {}`. 팀 표준에 "에러를 조용히 삼키지 말라"고 분명히 적었는데도, localStorage 접근부의 빈 catch는 그대로 남았어요. 타입 한 군데는 고치면서, 에러 처리 한 군데는 시드 패턴을 따라갔어요.

왜 이렇게 갈렸는지는 이 실험만으로 단정할 수 없어요. 다만 관찰된 사실은 분명해요. **지시 문서가 모든 안티패턴을 똑같이 막아주지는 않아요.**

## 실무에서 뭘 가져갈까

1. **코드베이스 품질이 에이전트 출력에 영향을 줄 수 있어요.** 에이전트가 자주 읽을 핵심 파일 — 엔트리포인트, 공용 컴포넌트, 자주 import되는 유틸 — 에 안티패턴이 반복돼 있지 않은지부터 봐요.

2. **CLAUDE.md에는 한 줄 슬로건 말고 구체적 기준을 적어요.** 이 실험에서 쓴 한 줄 지시("시니어처럼 모범 사례를 우선하라")는 거의 효과가 없었어요. "구체 타입을 써라, 에러를 삼키지 마라, 안정적 key를 써라"처럼 카테고리별로 적었을 때 재현율이 절반으로 떨어졌어요.

3. **지시만으로 다 막힌다고 믿지는 말아요.** 가장 효과가 좋았던 팀 표준에서도 재현율은 0이 아니라 10.6%였어요. 에러 처리 같은 일부 패턴은 지시를 줘도 절반은 빠져나갔어요. 빈 catch처럼 지시가 자주 놓치는 패턴은 lint 규칙이나 CI 검사를 함께 두는 방법을 고려할 수 있어요(이 실험이 그 효과를 직접 본 건 아니에요).

검증 가능한 한 줄로 정리하면 이래요. **CLAUDE.md에 팀 표준을 구체적으로 명시하면, 안티패턴이 반복 등장하는 코드베이스에서 에이전트가 같은 문제를 다시 만들 가능성이 줄어들어요** — 이 실험에서는 22.7%에서 10.6%로.

## 한계

방향성을 본 탐색적 실험이에요. 단일 스택(Vite + React + TypeScript)의 단일 코드베이스 위에서 했고, 조건당 15번 실행이에요. 다른 언어나 도메인, 더 큰 프로젝트에서도 같은 크기로 나타날지는 이 데이터로 말할 수 없어요. 수치는 "이 조건에서 이렇게 관찰됐다" 수준으로 읽어주면 돼요. 6종 중 일부(파생 상태 useEffect, 로직 중복 등)는 발현 횟수가 너무 적어 해석을 보류했어요.

또 이 실험이 변수로 둔 건 'CLAUDE.md 지시'뿐이에요. 작은 단일 코드베이스에서 한 번씩 생성했고, 에이전트 출력을 품질 기준으로 다시 검사하는 평가 단계는 적용하지 않았어요. 그래서 팀 표준의 10.6%나 빈 catch의 절반 잔존을 '지시의 일반적 한계'로 읽으면 안 돼요 — 더 큰 코드베이스나 그런 평가 단계를 더한 결과는 이 실험 밖이에요.
