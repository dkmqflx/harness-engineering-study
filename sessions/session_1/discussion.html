오늘의 발제

글을 다 못 읽고 와도 같이 이야기할 수 있게, 가볍게 던지는 질문들. 더 궁금하면 각 발제의 근거 보기를 펼치세요.

01

CLAUDE.md 규칙, 어디에 어떻게 쓸까

김도현

금지하는 말이 오히려 그 행동을 부른 적 있나요? 어떻게 바꿔 말하면 좋을까요?

아티클 근거 Do NOT add comments 지시가 오히려 주석을 유발했다. 부정형이 금지한 행동을 끌어내는 Pink Elephant 효과로 관찰됨.

규칙을 다른 파일로 빼두면 더 잘 지켜질까, 아니면 안 보고 지나칠까?

아티클 근거 @import 없이 docs/만 참조하면 파일을 읽지 않아 규칙 준수율이 떨어졌고, 토큰 절감도 없었다 (thin 조건 Task 1이 fat의 4.3배, 161,668 토큰).

규칙이 안 지켜질 때 — 규칙을 줄여야 할까, 늘 보이게 둬야 할까?

아티클 근거 Anthropic은 규칙 수를 줄이라고 권고한다. 반대 선택지로, 항상 로드되는 컨텍스트(큰 CLAUDE.md 또는 @import)에 두는 방법이 있다.
02

점수를 목표로 주면 UI를 망칠까

김찬영

점수 같은 목표를 주면 에이전트가 '점수만' 올리고 정작 중요한 걸 망칠 때가 있을까?

아티클 근거 Lighthouse 95점 이상만 제시하자 99.7점까지 올렸지만 사람이 보기 편한 UI를 망쳤다 (Goodhart's Law 양상).

목표를 아예 안 주는 게 더 나은 결과를 낼 때가 있을까? 왜 그럴까?

아티클 근거 점수 목표가 없던 시나리오 C(평균 92점)의 결과물이 오히려 더 나았다.

'보기 편함'처럼 숫자로 재기 어려운 품질을 에이전트에게 어떻게 지킬까?

아티클 근거 UI 보존 제약을 명시한 시나리오 D는 96.7점에 도달했다.
03

에이전트도 나쁜 코드 습관을 따라 할까

김시온

주변 코드가 지저분하면 에이전트도 그 나쁜 습관을 따라 할까요?

아티클 근거 더러운 코드베이스 위에서 새 기능을 시키면 깨끗한 코드베이스보다 안티패턴 재현율이 높아졌다 (6.1%→22.7%).

'이런 패턴은 쓰지 마'라고 콕 집어 말하면 정말 줄어들까요?

아티클 근거 콕 집은 사전 지시로 표면적 안티패턴은 줄었으나, 측정 항목과 지시 항목이 겹쳐 효과가 과장됐을 수 있다(동어반복 우려).

작성 뒤 스스로 점검하게 하면 나아질까요? 끝까지 안 고쳐지는 건 뭘까요?

아티클 근거 작성 후 자기점검은 재현율을 29%→15%로 낮췄지만, 가장 깊이 밴 패턴(디자인 토큰 등)은 100%→100%로 그대로였다.

교차 논제

세 실험을 가로지르는 질문

에이전트에 준 신호(목표·금지어·주변 코드)가 의도와 다른 행동을 부르는 공통 패턴이 있을까요? 미리 알아채려면?

세 글에서 단일 지표 목표 → UI 훼손 · 부정형 지시 → 금지 행동 유발 · 더러운 주변 코드 → 안티패턴 모방.

이런 실험 결과를 얼마나 믿을 수 있을까요? 무엇이 더 있으면 신뢰가 갈까요?

세 글에서 실행 횟수(태스크당 1회 / 15회 / 96회), 측정 자동화 여부, 채점자 독립성이 글마다 제각각이다.
